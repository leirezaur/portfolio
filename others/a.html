ux/ui design

arinc calendar exchange

client

Collins Aerospace - ARINCDirect

year

2021 -

company/school

Nuvolar

Collins Aerospace is one of the world’s largest suppliers of aerospace and aeronautics products, based in the USA. Even though it’s mainly focused on manufacturing and structures, Collins Aerospace's ARINCDirect is the most complete provider of business aviation solutions in the market today.

the goal 🎯

Flight departments worldwide (flight crew, dispatchers and operation managers) face some digital challenges nowadays. The software they use is very technical and broad, confusing to use and poorly connected to third parties. Collins’ purpose is to design and offer the best flight scheduling tool in the market to be the industry leaders worldwide and attract more customers and users in the following years.

The big challenge we were presented came with 3 main objectives to cover in the first MVP iteration:

  • Build a persona-focused product. Customers only want to pay for what they use and it’s also easier for them to have simple effective tools for each action.
  • Scalability. The product we create has to be ready for growth and future improvements and new features that will come.
  • Responsiveness. The Calendar Exchange tool will be built as a PWA for users to access anywhere. This is a huge step forward among competitors in the aviation market.

    • The scope of this project included a big discovery phase with research and user feedback of the current tools used, flow definition, wireframing, user testing, high-fidelity design and interactions delivery, development, Beta release and final public release live in April 2022. The brief was closed in March 2021 and had January 2022 as a deadline for the Beta version, which we successfully accomplished.

    My role in this Scheduling Engine project has been and currently is that of the UX/UI design lead.

    crafting the solution 💭

    Together with the Product Owner, the first months I run the investigation part of the project, having March as a deadline to close the low-fidelity designs. We first interviewed the stakeholders and more in depth the users, to get a demo and feedback on how they were working at the time.

    We generated a board of the user stories and basic actions they mentioned and realized the scope of the project was quite big. So the first task became the prioritization of the MVP, which we double checked with the stakeholders to look for approval. With the MVP user stories in mind, we then started mapping out connections, journeys and flows. The first stage included this main user stories (of course more):

  • As a flight dispatcher, I want to know the availability and agendas of my fleet of aircraft, so that I can plan ahead routes and events.
  • As a flight dispatcher, I want to be able to create new flight plans for my aircraft so that the quotation and scheduling process of crew can start running.
  • As a flight dispatcher, I want to be able to ‘travel in time’, so that I can access old data or plan future events according to AC availability.

    • A small benchmarking study was run to look for possible references in the market that already solve some of the user needs mentioned. Apps like Google Calendar and Apple Calendar came up to be very useful when it came to define timeline views, while even some flight booking apps’ features (Skyscanner or Southwest Airlines) served us for the trip building ideas, surprisingly.

    Having those in mind, the first low-fidelity wireframes were designed. They were continuously shown to all the stakeholders and users in interactive prototypes, to grow and enrich the first MVP to the fullest in the allocated time.

    The challenges faced along the way were multiple, turning out in the end as some excellent learning experiences. Some of the examples we had to discuss and redefine included features such as:

  • The position of the navigable calendar became a hot topic for some months. We finally chose the approach of keeping it as a modal, because the left panel proposal was stealing too much real state, specially in tablet. (The collapsable version was better adapted for future additional functionalities though, so thinking about scalability, it may come back in future iterations).
  • Although it looked like a good idea in the first drafts, the FAB button for the action of creating a new event in desktop didn’t work. We kept it for mobile as it eases usability, but it only caused some extra efforts in higher resolutions.

  • Users’ feedback was very valuable to design mobile solutions: dispatchers don’t usually work from portable devices, so its purpose is mainly focused on ‘read-only’.

  • The ability for users to create new trips, called Trip builder, is designed to be a collapsable menu, for the users to be able to navigate through the timeline while scheduling. We quickly discarded the option to direct users to a whole new screen for that reason, although it was the initial plan.
  • (Learning + work for the future) Users’ feedback made it clear that the usually navigate the app by tabbing in the keyboard and not using the mouse at all. However, we didn’t manage to fully cover this need in the calendar and time pickers of the form, due to the Local and UTC timeframe changes. This topic has been a bit of a nightmare and we are working hard on it to come up with a solution to satisfy all stakeholders, users and developers in the following release.

    • Once those were successfully closed and the low-fis approved by the end of March 2021, the workload focused on the high-fidelities and final deliverables of the app in all its resolutions. Collins already had a common Design system in place called Sonic, and this made the styling part easier to start. Having in mind the users’ journey, the app had been first designed in dark mode.

    We created new components and a small mini library to cover all the needed new assets and grew the DS with consistency and constant communication across teams, having weekly governance meetings with developers too for sync.

    final result ✨

    Here you can navigate through the final prototypes in desktop and mobile.

    The final metrics of this project will come live in some months, as it was just launched live, but so far the feedback got from the Beta users and stakeholders has been beyond positive as expected. This shows a clear good communication and expectation management we had through the whole process, and sets some actionable next steps to follow in the next iterations.

    This personally is my main takeaway from this project, being able to align so many stakeholders’ needs and cover complex requirements to build a functional top-notch tool. It’s been a whole experience to deal with so many different teams involved but all the constant and flowing communication has definitely paid off on the final result.

    We are currently working on the new enhancements and features of the tool to keep delivering solutions to the users’ feedback in the following iterations.
